Для запуска Tarantool-сервера нужно ввести одну из следующих команд:

$ **tarantool**
# ИЛИ
$ **tarantool** *опции*
# ИЛИ
$ **tarantool** *файл-инициализации-на-lua* **[** *аргументы* **]**

5.1. Опции командной строки¶

-h, --help¶

Вывести на экран список всех опций командной строки с комментариями.

-V, --version¶

Вывести на экран имя продукта и номер версии, например:

$ ./tarantool --version
Tarantool 1.7.0-1216-g73f7154
Target: Linux-x86_64-Debug
...

В нашем примере:

"Tarantool" — это имя асинхронной сетевой программной среды.

Номер версии составлен по стандартной схеме из 3 частей: <major>-<minor>-<patch>, где <major> изменяется очень редко, <minor> увеличивается на каждый стабильный релиз (milestone) и свидетельствует о возможных несовместимостях с предыдущими версиями, а <patch> указывает количество релизов с доработками с момента текущего стабильного релиза (milestone'а). Для сборок, не предназначенных для релиза, здесь могут быть добавлены также номер и SHA1-код коммита, которые указывают, насколько сильно текущая сборка отличается от предыдущего релиза.

"Target" — это платформа, на которой была сделана данная сборка Tarantool'а. За этой строчкой может следовать информация для данной платформы.

Примечание

Идентификатор версии (version id) в Tarantool'е получают с помощью команды git describe. По этому идентификатору можно в любой момент извлечь соответствующие исходные файлы из git-репозитория.

5.2. Универсальный код ресурса (URI)¶

Некоторые функции и параметры конфигурации в Tarantool'е зависят от универсального кода доступа (URI = Universal Resource Identifier). Формат URI-строки аналогичен общепринятому синтаксису URI-схемы и может содержать (перечисляем по порядку) имя пользователя для входа в систему, пароль, имя или IP-адрес хоста, номер порта. Обязательным параметром является только номер порта. Пароль обязателен, если указано имя пользователя (за исключением имени 'guest'). Таким образом, синтаксис URI формально выглядит так: [host:]port или [username:password@]host:port. Если не указан host, то значением по умолчанию является '0.0.0.0' (т.е. любой адрес формата IPv4) или '[::]' (т.е. любой адрес формата IPv6) на локальной машине. Если не указаны username:password, то значением по умолчанию является 'guest'. Вот несколько примеров:

Часть URI

Пример
port 3301
host:port 127.0.0.1:3301
username:password@host:port notguest:sesame@mail.ru:3301

В некоторых случаях вместо URI можно указывать сокет домена Unix, например "unix/:/tmp/unix_domain_socket.sock" или просто "/tmp/unix_domain_socket.sock".

См. также Appendix E: Cookbook, где приводится вариант реализации для парсинга URI-адресов.

5.3. Файл инициализации¶

Если в команде запуска Tarantool'а указан файл инициализации на языке Lua, то запуск начинается с вызова Lua-программы из файла с именем "script.lua". Эта Lua-программа может принимать дальнейшие аргументы, указанные в командной строке, или использовать функции операционной системы, такие как getenv(). Если после запуска Tarantool'а предполагается использовать сервер баз данных или требуется открыть порты на прослушивание, то Lua-программа почти всегда начинается с вызова box.cfg(). Для примера возьмем файл script.lua, который содержит следующие строки:

#!/usr/bin/env tarantool
box.cfg{
    listen              = os.getenv("LISTEN_URI"),
    slab_alloc_arena    = 0.1,
    pid_file            = "tarantool.pid",
    rows_per_wal        = 50
}
print('Starting ', arg[1])

Далее предположим, что системная переменная LISTEN_URI содержит значение 3301, а вызов командной строки таков: ~/tarantool/src/tarantool script.lua ARG. Вывод на экране в таком случае будет выглядеть следующим образом:

$ export LISTEN_URI=3301
$ ~/tarantool/src/tarantool script.lua ARG
... main/101/script.lua C> version 1.7.0-1216-g73f7154
... main/101/script.lua C> log level 5
... main/101/script.lua I> mapping 107374184 bytes for a shared arena...
... main/101/script.lua I> recovery start
... main/101/script.lua I> recovering from './00000000000000000000.snap'
... main/101/script.lua I> primary: bound to 0.0.0.0:3301
... main/102/leave_local_hot_standby I> ready to accept requests
Starting  ARG
... main C> entering the event loop

Если после инициализации вам понадобится запустить интерактивную Tarantool-сессию на том же терминале, то это можно сделать с помощью команды console.start().

5.4. Параметры конфигурации¶

Параметры конфигурации задаются следующим образом:
box.cfg{[key = value [, key = value ...]]}

Поскольку вызов box.cfg может содержать много параметров конфигурации, причем некоторые параметры (например, адреса директорий) могут со временем меняться, то имеет смысл хранить box.cfg в отдельном Lua-файле. Обычно это Lua-файл для инициализации, который указывается в командной строке при запуске Tarantool'а.

Большинство параметров конфигурации служат для настройки выделения ресурсов и открытия портов, а также для задания поведение базы данных. Все параметры являются необязательными. Некоторые параметры являются динамическими, т.е. их можно менять во время исполнения с помощью повторного вызова box.cfg{}.

Для просмотра все не нулевых значений параметров, введите box.cfg (без фигурных скобок). Для просмотра значения определенного параметра, например адреса для прослушивания, введите box.cfg.listen.

В следующих разделах описаны все параметры, которые служат для настройки основных операций, хранения, создания статических снимков данных, репликации, работы с сетевыми соединениями и журналирования.

5.4.1. Основные параметры¶

background¶

Запустить Tarantool-сервер в фоновом режиме. При этом нужно задать не нулевые значения для параметров logger и pid_file.

Тип значения: boolean
По умолчанию: false
Динамический: нет

coredump¶

Параметр устарел. Не используйте его.

Тип значения: boolean
По умолчанию: false
Динамический: нет

custom_proc_title¶

Добавить указанную строку к заголовку процесса Tarantool-сервера (заголовок процесса выводится в колонке COMMAND в ответ на команды ps -ef и top -c).

Например, информация о процессе Tarantool-сервера, которая выводится по команде ps -ef, обычно имеет следующий вид:

$ ps -ef | grep tarantool
1000     14939 14188  1 10:53 pts/2    00:00:13 tarantool <running>

Однако, если в конфигурации указан параметр custom_proc_title='sessions', то информация в выводе ps -ef будет другой:

$ ps -ef | grep tarantool
1000     14939 14188  1 10:53 pts/2    00:00:16 tarantool <running>: sessions

Тип значения: string
По умолчанию: null
Динамический: да

listen¶

Номер порта для чтения/записи данных, либо URI-строка (URI = Universal Resource Identifier). Значения по умолчанию у данного параметра нет, поэтому он обязателен, если у клиентов должна быть возможность устанавливать соединение с Tarantool-сервером по портам, не являющимся “портами для администрирования”. Соединения через listen=URI иногда называют соединениями по "бинарному протоколу" или по "главному порту".

Стандартное значение — это 3301. Порт для прослушивания можно задавать также для режима горячего резервирования.

Примечание

Сервер-реплика принимает запросы на соединение на этом же порту. Реплика принимает только запросы на чтение, пока ей не будет назначена роль главного сервера.

Тип значения: integer или string
По умолчанию: null
Динамический: да

pid_file¶

Сохранить идентификатор процесса в указанный файл. Путь можно указывать относительно рабочей директории (work_dir). Стандартное значение — это “tarantool.pid”.

Тип значения: string
По умолчанию: null
Динамический: нет

read_only¶

Перевести сервер в режим "доступен только на чтение". После установки этого параметра Tarantool-сервер будет возвращать ошибку ER_READONLY на все запросы на изменение данных.

Тип значения: boolean
По умолчанию: false
Динамический: да

snap_dir¶

Директория для хранения статических снимков данных (.snap-файлов). Путь можно указывать относительно рабочей директории (work_dir). Если этот параметр не задан, то по умолчанию используется рабочая директория (work_dir). См. также wal_dir.

Тип значения: string
По умолчанию: "."
Динамический: нет

vinyl_dir¶

Директория для хранения файлов и поддиректорий движка vinyl. Путь можно указывать относительно рабочей директории (work_dir). Если этот параметр не задан, то по умолчанию используется рабочая директория (work_dir).

Тип значения: string
По умолчанию: "."
Динамический: нет

username¶

UNIX-имя пользователя, на которое нужно переключиться после запуска.

Тип значения: string
По умолчанию: null
Динамический: нет

wal_dir¶

Директория для хранения журналов упреждающей записи (.xlog-файлов). Путь можно указывать относительно рабочей директории (work_dir). Иногда используют разные пути к wal_dir и snap_dir, чтобы .xlog-файлы и .snap-файлы можно было хранить на разных дисках. Если этот параметр не задан, то по умолчанию используется рабочая директория (work_dir).

Тип значения: string
По умолчанию: "."
Динамический: нет

work_dir¶

Директория для хранения рабочих файлов базы данных. После запуска сервер переключается на work_dir с помощью chdir(2). Путь можно указывать относительно текущей директории. Если этот параметр не задан, то по умолчанию используется текущая директория. Относительно work_dir можно указывать другие параметры. Например, если задать
box.cfg{work_dir='/home/user/A',wal_dir='B',snap_dir='C'}
то .xlog-файлы будут храниться в директории /home/user/A/B, .snap-файлы — в директории /home/user/A/C, а все прочие файлы и поддиректории — в рабочей директории /home/user/A.

Тип значения: string
По умолчанию: null
Динамический: нет

5.4.2. Настройка хранилища данных¶

slab_alloc_arena¶

Объем памяти (в гигабайтах), который Tarantool выделяет для хранения кортежей. Когда этот объем памяти оказывается занят, Tarantool-сервер начинает выдавать ошибку ER_MEMORY_ISSUE на запросы INSERT и UPDATE. Данное ограничение относится только к хранению кортежей: для хранения индексов и информации об установленных соединениях Tarantool-сервер использует дополнительную память. Таким образом, в зависимости от конкретной конфигурации и нагрузки, фактическое превышение расхода памяти у Tarantool-сервера может составлять до 20% от указанного здесь значения.

Тип значения: float
По умолчанию: 1.0
Динамический: нет

slab_alloc_factor¶

Данный параметр используется в качестве множителя при вычислении размера фрагментов памяти, выделяемых для хранения кортежей. В некоторых случаях (в зависимости от общего объема памяти и разброса по размеру кортежей) вы можете снизить объем выделенной памяти, задав меньшее значение множителя.

Тип значения: float
По умолчанию: 1.1
Динамический: нет

slab_alloc_maximal¶

Максимально возможный размер фрагмента памяти, выделяемого для одного кортежа. Можно задать больший размер, если необходимо хранить кортежи большого размера.

Тип значения: integer
По умолчанию: 1048576
Динамический: нет

slab_alloc_minimal¶

Минимально возможный размер фрагмента памяти, выделяемого для одного кортежа. Можно задать меньший размер, если большинство кортежей в базе имеют очень маленький размер. Значение может лежать в интервале от 8 до 1048280 включительно.

Тип значения: integer
По умолчанию: 16
Динамический: нет

vinyl¶

Пользовательские значения параметров для движка vinyl можно задать с помощью запроса следующего вида:

vinyl = {
  run_age_wm = number,
  run_age_period = number of seconds,
  memory_limit = number of gigabytes,
  compact_wm = number,
  threads = number,
  run_age = number,
  run_prio = number,
}

В будущем этот способ может быть пересмотрен.

Значения по умолчанию:

vinyl = {
  run_age_wm = 0,
  run_age_period = 0,
  memory_limit = 1,
  compact_wm = 2,
  threads = 5,
  run_age = 0,
  run_prio = 2,
}

5.4.3. Фоновая программа для работы со снимками¶

Фоновая программа для работы со снимками (snapshot daemon) — это постоянно запущенный файбер. Он может срабатывать через заданные промежутки времени, создавая новые файлы-снимки (.snap-файлы) и удаляя старые. При удалении файла-снимка удаляются также все WAL-файлы (.xlog), которые старше удаленного файла-снимка и содержат информацию, которая была в удаленном снимке.

Параметры snapshot_period и snapshot_count указывают длину интервалов для срабатывания фоновой программы и количество сделанных файлов-снимков, после которого можно удалять самый старый снимок.

snapshot_period¶

Интервал (в секундах) между срабатываниями фоновой программы. Если значение snapshot_period больше нуля, а также имели место какие-либо изменения в базе, то фоновая программа будет каждые snapshot_period секунд вызывать метод box.snapshot, и при каждом таком вызове будет создаваться новый снимок.

Например, в результате запроса box.cfg{snapshot_period=3600} фоновая программа будет раз в час создавать новый снимок.

Тип значения: integer
По умолчанию: 0
Динамический: да

snapshot_count¶

Максимальное количество снимков в директории snap_dir, после которого фоновая программа может удалять самые старые снимки. Если значение snapshot_count равно нулю, то фоновая программа не занимается удалением старых снимков. Например, в результате следующего запроса:

box.cfg{
    snapshot_period = 3600,
    snapshot_count  = 10
}

фоновая программа будет раз в час создавать новый снимок до тех пор, пока не создаст 10 снимков. После создания 11-го снимка фоновая программа удалит самый старый снимок и все связанные с ним WAL-файлы.

Тип значения: integer
По умолчанию: 6
Динамический: да

5.4.4. Журналирование и снимки в бинарном формате¶

panic_on_snap_error,
panic_on_wal_error,
rows_per_wal,
snap_io_rate_limit,
wal_mode,
wal_dir_rescan_delay
panic_on_snap_error¶

Прервать исполнение, если при чтении файла-снимка возникает ошибка (на этапе запуска Tarantool-сервера).

Тип значения: boolean
По умолчанию: true
Динамический: нет

panic_on_wal_error¶

Прервать исполнение, если при чтении WAL-файла возникает ошибка (на этапе запуска Tarantool-сервера, либо на этапе первого запроса данных со стороны реплики).

Тип значения: boolean
По умолчанию: true
Динамический: да

rows_per_wal¶

Максимальное количество записей, которые можно хранить в одном WAL-файле. Когда это ограничение достигнуто, то Tarantool создает новый WAL-файл и называет его по LSN-номеру первой записи в новом файле (<lsn-номер-первой-записи>.xlog). Это может пригодиться при простой организации создания резервных копий с помощью rsync.

Тип значения: integer
По умолчанию: 500000
Динамический: нет

snap_io_rate_limit¶

Уменьшить падение производительности на запросах INSERT/UPDATE/DELETE во время отработки box.snapshot с помощью ограничения на максимальное количество мегабайт, которое может записываться на диск в секунду. С той же целью можно развести директории wal_dir и snap_dir по разным адресам и сохранять снимки на другом диске.

Тип значения: float
По умолчанию: null
Динамический: да

wal_mode¶

Задать режим синхронизации между файберами, WAL-файлом и диском:

  • none: запись в WAL-файл не ведется;

  • write: файберы ждут, пока их данные будут записаны в WAL-файл (без fsync(2));

  • fsync: файберы ожидают свои данные, fsync(2) происходит после каждого вызова write(2);

Тип значения: string
По умолчанию: "write"
Динамический: да

wal_dir_rescan_delay¶

Количество секунд между периодическим сканированием директории с WAL-файлами на предмет наличия изменений в WAL-файлах, которые следует синхронизировать в рамках репликации или режима горячей готовности.

Тип значения: float
По умолчанию: 2
Динамический: нет

5.4.5. Репликация¶

replication_source¶

Источник репликации. Если в параметре replication_source указана не пустая строка, то такой Tarantool-сервер считается репликой. Сервер-реплика будет пытаться установить соединение с главным сервером, чей адрес указан в параметре replication_source в виде URI (Universal Resource Identifier), например konstantin:secret_password@tarantool.org:3301.

Если в кластере более одного источника репликации, то следует указать массив из URI, например
box.cfg{replication_source = {uri#1,uri#2}}

Если один из URI соответствует текущем серверу — т.е. тому, на котором сейчас выполняется запрос box.cfg{}, — то такой источник игнорируется. Таким образом, можно использовать одни и те же значения replication_source на многих серверах.

Имя пользователя по умолчанию — ‘guest’. Сервер-реплика не принимает запросы на изменение данных на порту для прослушивания (параметр listen). Параметр replication_source является динамическим: чтобы перевести реплику в режим главного сервера, нужно только задать пустую строку в replication_source и выполнить запрос box.cfg{replication_source=new-value}.

Тип значения: string
По умолчанию: null
Динамический: yes

5.4.6. Работа с сетевыми соединениями¶

io_collect_interval,
readahead
io_collect_interval¶

Между итерациями основного цикла Tarantool-сервер уходит в спящий режим на io_collect_interval секунд. Этот параметр можно использовать для ограничения нагрузки на процессор в том случае, когда количество клиентских соединений очень велико, но запросы от них приходят нечасто (например, от каждого соединения приходит всего несколько запросов в секунду).

Тип значения: float
По умолчанию: null
Динамический: да

readahead¶

Размер буфера предварительного чтения на каждое клиентское соединение. Чем больше размер буфера, тем больше памяти будет потреблять каждое активное соединение и тем больше запросов можно будет зачитать из системного буфера за один системный вызов. Общее правило для определения оптимального размера буфера таково: буфер должен вмещать по крайней мере несколько десятков запросов. Поэтому, если средний размер кортежа в запросе у вас большой, например несколько кило- или даже мегабайт, то размер буфера предварительного чтения следует увеличить. Если вы не используете пакетную обработку запросов, то лучше оставить размер буфера по умолчанию.

Тип значения: integer
По умолчанию: 16320
Динамический: да

5.4.7. Запись в журнал¶

log_level¶

Уровень детализации при записи в журнал. Всего есть шесть уровней:

  • 1 – SYSERROR
  • 2 – ERROR
  • 3 – CRITICAL
  • 4 – WARNING
  • 5 – INFO
  • 6 – DEBUG

С помощью настройки уровня детализации можно включать журналирование всех классов событий выше или ниже указанного уровня. По умолчанию Tarantool выводит свой журнал в стандартный поток ошибок (stderr), но вывод ошибок можно настроить с помощью параметра logger.

Тип значения: integer
По умолчанию: 5
Динамический: да

logger¶

По умолчанию Tarantool выводит свой журнал в стандартный поток ошибок ( stderr). С помощью данного параметра вывод ошибок можно перенастроить на вывод в файл, в программный канал (pipe) или в системный регистратор (system logger).

Пример настроек:

box.cfg{logger = 'tarantool.log'}
-- or
box.cfg{logger = 'file: tarantool.log'}

В результате в Tarantool-директории, используемой по умолчанию, будет открыт на запись файл tarantool.log. Если префикс в значении параметра logger не указан или же указан префикс "file:", то дальнейшая строка интерпретируется как путь к файлу.

Пример настроек:

box.cfg{logger = '| cronolog tarantool.log'}
-- или
box.cfg{logger = 'pipe: cronolog tarantool.log'}'

В результате при запуске Tarantool-сервера будет также запущена программа cronolog, и все сообщения для журнала буду пересылаться на стандартный ввод (stdin) программы cronolog. Если значение параметра logger начинается с символа '|' или с префикса "pipe:", то дальнейшая строка интерпретируется как Unix-конвейер.

Пример настроек:

box.cfg{logger = 'syslog:identity=tarantool'}
-- или
box.cfg{logger = 'syslog:facility=user'}
-- или
box.cfg{logger = 'syslog:identity=tarantool,facility=user'}

Если значение параметра logger начинается с префикса "syslog:", то дальнейшая строка интерпретируется как сообщение для программы syslogd, которая обычно запущена в фоновом режиме на любой Unix-платформе. В этой строке можно задать значения identity, facility или оба эти значения сразу. В качестве identity может быть задана любая строка (значение по умолчанию — tarantool), которая будет использоваться как префикс перед всеми сообщениями в журнале. В качестве facility задается аббревиатура имени одного из типов ПО в программе syslog (значение по умолчанию — user), которая указывает программе syslogd, куда/кому передавать сообщения.

Возможные значения facility: auth, authpriv, cron, daemon, ftp, kern, lpr, mail, news, security, syslog, user, uucp, local0, local1, local2, local3, local4, local5, local6, local7.

На данный момент значение``facility`` игнорируется, но в будущем оно будет использоваться.

При записи сообщений в файл Tarantool переоткрывает файл с журналом на каждый SIGHUP. Если журнал является программой, то ее pid-идентификатор сохраняется в переменной log.logger_pid. Для ротации журнала нужно послать сигнал программе с этим pid-идентификатором.

Тип значения: string
По умолчанию: null
Динамический: нет

logger_nonblock¶

Если в значении параметра logger_nonblock указано true, то Tarantool не блокирует файл с журналом на запись, и если очередь на запись в ядре оказывается переполнена, то последующие сообщения от Tarantool'а не попадают в очередь и теряются, пока очередь не очистится. Если в параметре log_level указан высокий уровень детализации и Tarantool отправляет в очередь на запись много сообщений для журнала, то можно увеличить скорость записи в журнал, выставив значение параметра logger_nonblock в true и пожертвовав некоторым количеством сообщений, которые при этом могут быть потеряны.

Тип значения: boolean
По умолчанию: true
Динамический: нет

too_long_threshold¶

Если обработка запроса занимает больше времени, чем указано в этом параметре (в секундах), то в журнал добавляется предупредительная запись. Это происходит, только если уровень детализации, указанный в параметре log_level, выше или равен 4 (WARNING).

Тип значения: float
По умолчанию: 0.5
Динамический: да

Пример работы с журналом:

В этом примере мы покажем, как работает ротация журнала, т.е. что происходит, когда настроена запись в журнал и используются сигналы для архивирования записей.

Запустите два отдельных терминала, Terminal #1 и Terminal #2.

В первом терминале (Terminal #1): запустим интерактивную Tarantool-сессию, настроим запись сообщений в журнал Log_file, а затем запишем в журнал сообщение "Log Line #1":

box.cfg{logger='Log_file'}
log = require('log')
log.info('Log Line #1')

Во втором терминале (Terminal #2): с помощью команды mv переименуем файл журнала в Log_file.bak. В результате следующее сообщение будет записано уже в файл Log_file.bak. 

mv Log_file Log_file.bak

В первом терминале (Terminal #1): запишем в журнал сообщение "Log Line #2". 

log.info('Log Line #2')

Во втором терминале (Terminal #2): с помощью команды ps выясним идентификатор процесса Tarantool-сервера.

ps -A | grep tarantool

Во втором терминале (Terminal #2): с помощью команды kill -HUP пошлем Tarantool-серверу SIGHUP-сигнал.

kill -HUP process_id

В первом терминале (Terminal #1): запишем в журнал сообщение "Log Line #3".

log.info('Log Line #3')

Во втором терминале (Terminal #2): с помощью команды less просмотрим содержимое файлов. Для файла Log_file.bak будут выведены следующие строки (если вы выполняете этот пример у себя, то отличаться должны только дата и время):

2015-11-30 15:13:06.373 [27469] main/101/interactive I> Log Line #1`
2015-11-30 15:14:25.973 [27469] main/101/interactive I> Log Line #2`

а для журнала Log_file вывод будет следующим:

log file has been reopened
2015-11-30 15:15:32.629 [27469] main/101/interactive I> Log Line #3

5.5. Режим горячего резервирования¶

Режим горячего резервирования позволяет организовать восстановление после сбоя простым образом, без настройки репликации. Для этого запустите на том же компьютере второй экземпляр Tarantool-сервера с теми же настройками box.cfg, включая директории и не нулевые URI-адреса. В случае успеха должно появиться предупреждение:

W> primary: [URI] is already in use, will retry binding after [n] seconds

Это означает, что второй экземпляр готов заменить первый, если тот выйдет из строя.

Режим горячего резервирования подразумевает, что на одном компьютере запущены два экземпляра Tarantool-сервера с одинаковой конфигурацией. Первый запущенный экземпляр является "главным" (primary), а второй — "резервным" (standby). Резервный экземпляр при запуске попытается установить соединение на порту для прослушивания, но у него не получится это сделать, поскольку адрес уже занят главным экземпляром. Таким образом, резервный экземпляр повторяет в цикле следующие действия: зачитывает данные из WAL-файла, который обновляется главным экземпляром (и благодаря этому оба экземпляра всегда синхронизированы между собой) и пытается установить соединение на порту для прослушивания. Если по какой-либо причине главный сервер выходит из строя, то порт для прослушивания освобождается, резервный экземпляр устанавливает на нем соединение и становится главным. Таким образом, время недоступности в случае сбоя оказывается почти незаметным.

При использовании горячего резервирования (значение параметра local_hot_standby равно "true") значение параметра wal_mode не должно быть равно "none".
